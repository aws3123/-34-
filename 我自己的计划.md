# 我的计划书

## 任务定位
我负责这个项目的**骨架 owner + 集成负责人**。

这不是去抢别人模块里的具体业务实现，而是确保整个仓库始终保持：

- 解决方案结构稳定
- 项目引用关系清晰
- 公共接线点不混乱
- 各成员的代码能顺利合并
- 主分支始终可构建、可测试、可继续开发

简单说，我的职责是：**先把路修好，再把各模块接起来，并在集成阶段做最后一道结构与质量把关。**

## 我的核心目标

### 1. 维护项目骨架稳定
保证以下结构长期保持清晰：

- `ISO11820.sln`
- `src/ISO11820.App`
- `src/ISO11820.Core`
- `tests/ISO11820.Tests`

### 2. 维护公共接线点
把所有跨模块共用的启动、配置、上下文、项目引用、构建设置控制在少数稳定入口里，避免每个人都去随手改启动流程。

### 3. 做模块集成与边界守卫
当其他 4 位成员开始推进各自模块后，我负责检查：

- UI 没有把 SQL 写进去
- Runtime 没有依赖 WinForms
- Persistence 没有反向依赖 UI
- Export / TestRecord 没有把逻辑散进窗体事件

### 4. 做阶段性验收
在每一轮开发后，负责统一执行 build / test / 启动验证，确认整个仓库仍然健康。

## 任务拆分

### 一、维护解决方案与项目结构
负责：

- 保持 `sln/csproj` 干净可用
- 维护项目引用关系
- 控制目标框架、输出设置、公共构建设置
- 补齐必要的全局忽略规则

### 二、维护启动与组合根
负责：

- `Program.cs`
- `App/Bootstrapper.cs`
- `App/Iso11820AppContext.cs`

要求：

- 各模块通过清晰入口挂进应用
- 启动流程不要分散在多个地方
- 配置、数据库初始化、运行时启动顺序明确

### 三、维护共享模型与跨模块契约
负责：

- `Shared/Models/**`
- `Shared/Events/**`
- 必要的配置模型

要求：

- 共享对象字段命名稳定
- 不在共享模型里混入 UI 控件或数据库连接对象
- 跨模块对接尽量通过共享快照/事件/结果对象完成

### 四、做集成测试与结构验收
每次合并别人的工作后，我负责统一验证：

- 是否还能 build
- 是否还能 test
- WinForms 外壳是否还能启动
- 目录边界是否被破坏
- 是否出现循环依赖或职责错位

### 五、管理阶段性协调
在 4 位协作者之间，我负责检查这些关键接口是否对齐：

- Runtime owner 提供的广播快照，UI 能否直接消费
- Persistence owner 提供的查询/保存入口，History / TestRecord 是否能接
- Export owner 的输出入口，是否不依赖窗体代码
- 各方新增的配置项，是否统一进 `appsettings.json`

## 完成标准
满足以下条件，才算我这条线完成得合格：

- `ISO11820.sln` 始终可加载、可构建
- 3 个项目的职责边界没有被打穿
- 各模块都通过稳定入口挂接到应用，而不是在窗体或随机工具类里硬连
- 共享模型足够支撑协作，但没有膨胀成大杂烩
- 别人的分支合并后，我可以较快恢复到 build/test 通过状态
- 关键结构约束仍成立：
  - `Core` 不依赖 WinForms/SQLite
  - `UI` 不直接写 SQL
  - `Infrastructure` 不反向依赖 `UI`

## 验证方式

### 1. 构建验证
运行：

```bash
"/c/Program Files/dotnet/dotnet.exe" build "ISO11820.sln"
```

要求：

- 解决方案整体 build 通过
- 不出现因为项目引用或目标框架导致的编译错误

### 2. 测试验证
运行：

```bash
"/c/Program Files/dotnet/dotnet.exe" test "tests/ISO11820.Tests/ISO11820.Tests.csproj"
```

要求：

- 所有现有测试通过
- 新增接线/共享契约变更不会打爆已有测试

### 3. 启动验证
运行：

```bash
"/c/Program Files/dotnet/dotnet.exe" run --project "src/ISO11820.App/ISO11820.App.csproj"
```

要求：

- 应用能正常启动
- 主窗体能打开
- 启动接线没有明显异常

### 4. 结构验证
人工检查：

- `Program.cs` 不变成业务逻辑堆放区
- `Bootstrapper.cs` 仍是统一接线点
- `Shared/**` 只放共享契约，不放大段业务逻辑
- 各 owner 没有大面积越界改别人模块

### 5. 集成验证
每轮合并后检查：

- `git status` 是否干净可控
- 是否新增了不该提交的临时文件
- 是否有人把 `bin/obj/.user` 之类产物带进仓库
- 是否出现跨层直接调用导致边界失守

## 我的交付物
- 稳定的 solution / project 结构
- 稳定的启动与组合根
- 稳定的共享契约与配置入口
- 阶段性集成后的可构建、可测试仓库
- 必要的集成/冒烟测试

## 我的工作原则
1. 不替别人“顺手做完”他的模块，而是保护边界、接好接口。
2. 先保证整体健康，再追求局部功能推进。
3. 出现冲突时，优先恢复 `build + test + run` 三件事。
4. 公共入口宁可少而稳，也不要到处都能改。
