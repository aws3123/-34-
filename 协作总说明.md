# 协作总说明

## 文档目的
这份文档用于统一当前 5 人协作开发方式，避免出现：

- 多人同时改同一批公共文件
- UI、运行时、数据库、导出逻辑互相串层
- 每个人都能跑一部分，但合起来无法构建
- 做完功能后不知道该如何联调与验收

本说明是对以下 5 份个人计划书的上位约束与补充：

- `任务计划-chaoxingstar.md`
- `任务计划-Nlpexi-e.md`
- `任务计划-userswdrs.md`
- `任务计划-Y-Yunye.md`
- `我自己的计划.md`

个人计划书定义“谁负责什么”，本说明定义“大家怎么一起工作”。

## 三、绝对边界规则

这是整个项目最重要的协作约束。

### 1. Core 不依赖外部实现
`src/ISO11820.Core/**` 中禁止出现：

- `System.Windows.Forms`
- `Microsoft.Data.Sqlite`
- 文件路径拼接作为基础设施实现细节
- `Control` / `Form` / `Invoke`

### 2. UI 不直接写业务核心逻辑
`src/ISO11820.App/UI/**` 中禁止出现：

- SQL 语句
- 状态机主判断
- 导出路径拼接
- `flag == "10000000"` 这类持久化规则散落在窗体事件中

### 3. Persistence 不反向依赖 UI
`src/ISO11820.App/Infrastructure/Persistence/**` 中禁止依赖：

- `UI/**`
- `Form`
- 任意控件对象

### 4. Export / TestRecord 不把逻辑塞进 Dialog
`TestRecordDialog.cs` 只负责输入与触发动作；
真正的保存与导出组织逻辑必须在：

- `Features/TestRecord/**`
- `Features/Export/**`
- `Infrastructure/FileStorage/**`

### 5. 共享契约必须稳定
跨模块交互优先通过以下层处理：

- `Shared/Models/**`
- `Shared/Events/**`
- `Config/**`
- `App/**`

不要让某个模块为了图快直接 new 另一个模块内部类再穿透调用。

## 五、接口对接规则

### 1. 运行时 → UI
由运行时 owner 输出统一快照，UI owner 只消费：

- `RuntimeSnapshot`
- `DataBroadcastEventArgs`

UI 不自己计算状态流转，不自己拼运行时温度结构。

### 2. 持久化 → 记录保存 / 查询 / 校准
由持久化 owner 提供统一访问入口：

- 历史记录查询入口
- 校准记录读写入口
- 试验记录落库入口

记录保存 owner 不直接把 SQL 写进 `Dialog` 或文件输出类。

### 3. 配置 → 所有模块
所有路径、数据库位置、输出目录、仿真参数统一从：

- `appsettings.json`
- `Config/**`

读取。

禁止在各自模块里随手硬编码新路径。

### 4. 导出 → UI
导出 owner 提供调用入口，UI 只负责：

- 收集当前要导出的试验标识
- 触发导出动作
- 显示成功/失败结果

## 六、推荐开发顺序

为了减少互相等待，推荐按下面顺序推进。

### 第 1 阶段：打牢公共底座
负责人：骨架 owner + 运行时 owner + 持久化 owner

目标：

- 解决方案、项目、引用关系稳定
- 配置可读取
- 数据库可初始化
- 运行时广播契约可用

### 第 2 阶段：UI 外壳 + 运行时主链路
负责人：UI owner + 运行时 owner

目标：

- 主界面能显示当前状态和温度
- `DataBroadcast` 能驱动 UI 更新
- 线程切换安全

### 第 3 阶段：新建试验 + 数据落点
负责人：UI owner + 持久化 owner

目标：

- 新建试验输入流程可走通
- 测试基础数据可保存
- 后续运行与查询有可依赖的数据结构

### 第 4 阶段：完成记录 + CSV + 导出
负责人：记录/导出 owner + 持久化 owner + 运行时 owner

目标：

- 试验完成后能记录
- CSV 路径和文件输出稳定
- 导出入口可用
- 保存后闭环成立

### 第 5 阶段：历史查询 + 校准
负责人：持久化 owner + UI owner

目标：

- 历史查询可展示
- 校准记录可查询/写入

### 第 6 阶段：全链路联调
负责人：骨架 owner 主导，全员参与

目标：

- 登录
- 新建试验
- 升温/就绪/记录/完成
- 保存试验记录
- 查询历史
- 导出文件

## 七、联调顺序

每次联调不要一次性全员一起改，按链路接入：

### 联调 1：启动与广播
- 骨架 owner
- 运行时 owner
- UI owner

验收：
- 应用能启动
- 广播能到 UI
- 无跨线程异常

### 联调 2：数据库与基础数据
- 骨架 owner
- 持久化 owner

验收：
- 数据库创建成功
- 初始账号/设备/传感器数据可写入

### 联调 3：新建试验与主流程
- UI owner
- 运行时 owner
- 持久化 owner

验收：
- 新建试验后主流程可进入运行态

### 联调 4：完成记录与导出
- 记录/导出 owner
- 持久化 owner
- 运行时 owner

验收：
- 完成后记录可保存
- CSV 路径正确
- 导出入口能接到试验标识

### 联调 5：全流程验收
- 全员

验收：
- 全链路可跑通
- build/test/run 正常

## 八、提交前最低验证要求

每个人在提交自己负责的模块前，至少完成以下检查。

### 1. 构建检查
```bash
"/c/Program Files/dotnet/dotnet.exe" build "ISO11820.sln"
```

### 2. 测试检查
```bash
"/c/Program Files/dotnet/dotnet.exe" test "tests/ISO11820.Tests/ISO11820.Tests.csproj"
```

### 3. 启动检查（涉及 UI / 接线变更时）
```bash
"/c/Program Files/dotnet/dotnet.exe" run --project "src/ISO11820.App/ISO11820.App.csproj"
```

### 4. Git 检查
```bash
git status --short
```

确保没有误带：

- `bin/`
- `obj/`
- `*.user`
- 临时测试文件
- 无关截图或本地缓存

## 九、验收口径

一个模块“写完了”不等于“完成了”。

必须同时满足：

1. 在自己的文件边界内完成主要目标。
2. 没有明显越界破坏别人的模块。
3. 能通过 build。
4. 相关 test 通过。
5. 该模块能和上下游完成最小联调。

例如：

- 运行时 owner 写完状态机，但 UI 一接就崩，不算完成。
- UI owner 画完界面，但把 SQL 写进 `MainForm`，不算完成。
- 持久化 owner 能建库，但表结构不符合规格，不算完成。
- 导出 owner 能生成文件，但路径不符合约定，也不算完成。

## 十、发生冲突时的处理顺序

如果多人改动冲突，按以下优先级处理：

### 优先级 1：恢复可构建
先让 `build` 恢复通过。

### 优先级 2：恢复共享契约稳定
优先确认：

- `Shared/Models/**`
- `Shared/Events/**`
- `Config/**`
- `App/**`

这些文件的最终口径。

### 优先级 3：恢复主链路
至少保证：

- 应用可启动
- 主界面可开
- 广播可到 UI

### 优先级 4：再恢复局部功能
比如历史查询、导出细节、校准等。

---

## 十一、最终协作原则

1. **谁的 owner 范围，谁先做主。**
2. **公共文件少动，动之前先沟通。**
3. **先保结构，再保功能，再做细节。**
4. **联调时以真实 build/test/run 结果为准，不以“我本地单文件没问题”为准。**
5. **任何人都不要把临时修复变成长期耦合。**

---

## 十二、当前阶段的统一目标

当前不是追求一次性做完整业务，而是先把这件事做稳：

> 让 5 个人可以在清晰边界下并行推进，并且任何一个阶段结束后，仓库都仍然是可构建、可测试、可继续集成的。

如果这个目标守住了，后面的功能扩展才会越做越顺；如果这个目标没守住，后面每多一个功能，协作成本都会翻倍。
